# Avatar Puppet — 設計書

## 1. プロダクトビジョン

「1枚の絵が、画面の向こうで生きているように感じられる体験」を、ブラウザだけで実現する。

ユーザーが描いた（または AI で生成した）キャラクターが、ボタン操作ひとつで笑い、怒り、手を振り、話しかけてくる。
「この子が動いている」「この子と話している」と感じられる体験がこのプロダクトのゴール。

## 2. ターゲットユーザー

AI 画像生成（Midjourney, Nano Banana 等）でキャラクターを作っている個人クリエイター。
自分が生成したキャラに命を吹き込みたいが、Live2D のリギングを学ぶ時間やコストはかけたくない人。

### 利用シーン

- キャラクター対話ページとして Web に公開
- 配信のオーバーレイ素材として使用
- ゲームの NPC 対話画面のプロトタイプ

## 3. ゴール定義（MVP）

**一文で言うと：**
1枚のキャラ画像をアップロードし、表情・ポーズ・セリフをボタンで切り替えると、キャラがぬるぬる動いて吹き出しで話す Web アプリが GitHub Pages で公開されている。

### 完了条件

| # | 条件 | 検証方法 |
|---|------|---------|
| G1 | PNG/JPEG/WebP 画像をアップロードできる | 3種類のファイルで確認 |
| G2 | キャラがパーツごとに分離されて画面中央に表示される | 頭・目・口・腕が独立して見える |
| G3 | 表情ボタン（最低4種）でスムーズに表情が変化する | 0.3秒以内にアニメーション開始、0.5秒以内に完了 |
| G4 | ポーズボタン（最低3種）で体が動く | 同上 |
| G5 | セリフボタンで吹き出し + 表情が連動する | 同時に発生 |
| G6 | 30fps 以上で滑らかに動く | Chrome DevTools FPS メーターで確認 |
| G7 | idle 時にキャラが呼吸している | 体が上下に微動 |
| G8 | GitHub Pages で公開されている | URL アクセスで動作確認 |
| G9 | スマホでも操作・表示が正常 | iPhone Safari, Android Chrome |

### ユーザー体験フロー

ページを開く → サンプルキャラが呼吸している
画像をアップロード → キャラが差し替わる
表情ボタンを押す → 目・口がスムーズに変化
ポーズボタンを押す → 腕・体が動く
セリフボタンを押す → 吹き出し + 表情連動 → 「話してる感」が生まれる ★ゴール

| ボーンスキニング | MeshPlane + Transform で十分。Stage C |

## 7. パーツ構造

1枚のキャラ画像を以下の矩形パーツに分割する。

┌─────────────────────┐ │ ┌───────────┐ │ │ │ head │ │ │ │ ┌──┐ ┌──┐│ │ │ │ │EL│ │ER││ │ │ │ └──┘ └──┘│ │ │ │ ┌────┐ │ │ │ │ │mouth│ │ │ │ │ └────┘ │ │ │ └───────────┘ │ │ ┌──┐┌─────────┐┌──┐│ │ │AL││ body ││AR││ │ │ ││ ││ ││ │ └──┘└─────────┘└──┘│ └─────────────────────┘


各パーツの属性：

- **region** — 元画像上の矩形座標（x, y, w, h）
- **anchor** — 回転中心（0〜1 正規化座標）。腕なら肩、頭なら首の付け根
- **zIndex** — 描画の重なり順。ポーズによって動的に変わる
- **meshGrid**（任意） — 変形させたいパーツのみ MeshPlane を使い、グリッド分割数を指定

変形不要なパーツは Sprite で軽量描画。目・口など変形させたいパーツのみ MeshPlane を使用。

## 8. アニメーション仕様

### 補間方式

毎フレーム、各パーツの現在の Transform 状態をターゲット状態に向かって easeInOut で補間する。

currentValue += (targetValue - currentValue) * smoothFactor * deltaTime


smoothFactor は 0.08〜0.12。ボタン押下から 0.3〜0.5 秒で遷移完了。

### idle（呼吸）モーション

常時、体パーツに sin 波の微小変動を加える。

body.y += sin(time * 2) * 1.5 body.scaleY = 1.0 + sin(time * 2) * 0.005 head.y += sin(time * 2 + 0.5) * 1.0


### 表情変化

目・口の scaleX, scaleY, rotation, position をプリセットに従って変更。

### ポーズ変化

腕の rotation と position、体と頭の rotation を変更。zIndex もポーズに連動。

## 13. 設計判断の根拠

**なぜ PixiJS か：** 2D WebGL 描画に特化。MeshPlane で頂点変形が可能。earcut を内蔵。軽量で高速。

**なぜ MeshPlane であって自前三角分割ではないか：** PixiJS の MeshPlane が矩形グリッドメッシュを内蔵提供する。MVP のパーツは矩形に近いため十分。earcut による自由形状メッシュは Stage C で必要になった時に導入する。

**なぜ 2ファイルか：** 500行規模のプロジェクトにディレクトリ構造は過剰。ファイル分割は300行を超えた時に行う。ロジック（main.ts）とデータ（presets.ts）の分離だけで十分。

**なぜ PWA か：** 既存コードの変更ゼロで vite-plugin-pwa を足すだけ。App Store 公開が必要になったら Capacitor でラップする（コード変更なし）。

**なぜ N-Sprite を直接採用しないか：** 実験段階（Star 22, Commits 19）で外部 AI API 依存。手動パーツ定義で確実に動く MVP を作り、Stage B で統合パスを用意する。

**なぜ Live2D / Spine を使わないか：** Live2D は独自ライセンス（OSS ではない）。Spine は有料（$69〜）。今回は OSS で完結させる。Inochi2D-ts は発展途上（Commits 14）でリスクが高い。
